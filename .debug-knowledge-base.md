# Debug Knowledge Base

## sync-to-user-claude repo-side deletion (2026-07-04)

**Symptom:** Files in a tracked root-level support directory were deleted from the repo working tree. Reproduced 3 times in one session. Caught via `git status --short`, restored via `git checkout --`.

**Root cause:** A support-directory symlink under `~/.claude/skills/` pointed into the repo. Any destructive operation (rmtree, unlink) that traverses through a parent-level symlink into the repo deletes tracked source files. The most dangerous code path: `_ensure_symlink` calls `shutil.rmtree(dst)` when `dst.is_dir()` and not `dst.is_symlink()`. If a parent directory (e.g., `~/.claude/skills`) is itself a symlink to `repo/skills/`, then `dst` resolves into the repo even though `dst` itself is not a symlink. The `is_symlink()` check on the final path component misses parent-level symlink traversal.

**Prior partial fix (PR #762):** Added the support directory to the `root_dirs` allowlist so the stale-cleanup loop preserves its symlink. This prevented the SYMLINK from being removed but did not prevent operations that traverse INTO the symlinked directory from deleting repo files.

**Fix:** Added `_resolves_inside(path, repo_root)` guard function. Applied before every `rmtree`, `unlink`, and `rmdir` in:
- `_ensure_symlink` (all destructive branches including race recovery)
- `_sync_skills_flat_symlinks` stale-cleanup loop
- `_main_inner` deferred stale-cleanup loop

When the guard detects that a path resolves inside the repo root, the operation is skipped with a loud stderr warning. Fail-safe: skip + warn beats guessing.

**Tests:** 9 new tests in `TestResolvesInside` (5) and `TestRepoSideDeletionGuard` (4). Key test: `test_ensure_symlink_blocks_rmtree_through_parent_symlink` — simulates `~/.claude/skills -> repo/skills/` and verifies `_ensure_symlink` refuses to rmtree the repo directory.

**Lesson:** Symlinks from the mirror (`~/.claude/`) into the repo create a trust-boundary violation. `Path.is_symlink()` only checks the final component; parent-level symlinks are invisible to it. Defense-in-depth: resolve the full path and check against the repo root before any destructive operation, regardless of which code path reaches the deletion.

## sync-to-user-claude repo-side deletion REOPENED (2026-07-05)

**Symptom:** 4th occurrence. Files in the support directory were deleted again WITH the `_resolves_inside` guard merged and deployed. Guard was observed blocking other paths (e.g., `research/content`) on a prior run. Files restored at 10:33:52 today.

**Analysis:** Exhaustive code analysis of all deletion sites in `sync-to-user-claude.py` found every `rmtree`, `unlink`, and `rmdir` is guarded by `_resolves_inside`. The support directory IS in `expected_names` (via `root_dirs` allowlist), so the stale-cleanup loop in `_sync_skills_flat_symlinks` skips it. Python 3.12's `Path.rglob("*")` does NOT descend into symlinked directories, so the deferred stale cleanup cannot reach files inside symlinked dirs. No other hook or script deletes these files.

**Residual path identified (hypothesis with reproduction test):** When a worktree session bypasses `_is_git_worktree` detection (e.g., `.git` file unreadable + `git rev-parse` failure), sync runs with `repo_root = worktree_path`. If the worktree branch predates a root-level support directory, that directory is absent from the worktree's `skills/`, so it is NOT added to `expected_names`. The stale cleanup then sees the support-directory symlink as stale, and `_resolves_inside(symlink, worktree_root)` returns False (the symlink resolves into the MAIN repo, not the worktree). The symlink is deleted. Reproduced in `TestProtectedRootsWorktreeScenario::test_worktree_sync_without_canonical_deletes_symlink`.

**Fix (3 layers):**
1. **`_is_git_worktree` safe default:** When `.git` is a file but unreadable AND `git rev-parse` fails, return True (assume worktree, refuse sync). Previously returned False (allowed sync).
2. **Multi-root `_resolves_inside`:** Accepts `Path | list[Path]`. In `_main_inner`, `protected_roots` is built from CWD-based `repo_root` PLUS the canonical `toolkit_path` from `~/.claude/.install-manifest.json`. Symlinks into the main repo are blocked even when `repo_root` is a worktree path.
3. **Codex mirror parity:** Added the support directory to `_codex_root_utility` (it had been treated as a category dir instead of a utility dir).

**Tests:** 12 new tests across 4 classes:
- `TestResolvesInsideMultiRoot` (4): list-of-roots matching, symlink-resolved-against-canonical
- `TestCanonicalRepoRoot` (3): manifest read, missing, nonexistent dir
- `TestWorktreeDetectionSafeDefault` (3): unreadable .git, submodule, worktree
- `TestProtectedRootsWorktreeScenario` (2): reproduces bug without fix, verifies fix

**Lesson:** `_resolves_inside(path, repo_root)` is only as good as `repo_root`. When the guard's reference point is wrong (worktree CWD instead of main repo), it silently passes. Defense: anchor the guard to a CANONICAL root from the install manifest, not just CWD. Also: `_is_git_worktree` must fail-safe to True when uncertain -- a false positive (blocking submodule sync) is harmless; a false negative (allowing worktree sync) causes data loss.
