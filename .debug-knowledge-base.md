# Debug Knowledge Base

## sync-to-user-claude repo-side deletion (2026-07-04)

**Symptom:** `skills/voice-shared-references/*.md` deleted from the repo working tree (tracked files). Reproduced 3 times in one session. Caught via `git status --short`, restored via `git checkout --`.

**Root cause:** `~/.claude/skills/voice-shared-references` is a symlink pointing into the repo. Any destructive operation (rmtree, unlink) that traverses through a parent-level symlink into the repo deletes tracked source files. The most dangerous code path: `_ensure_symlink` calls `shutil.rmtree(dst)` when `dst.is_dir()` and not `dst.is_symlink()`. If a parent directory (e.g., `~/.claude/skills`) is itself a symlink to `repo/skills/`, then `dst` resolves into the repo even though `dst` itself is not a symlink. The `is_symlink()` check on the final path component misses parent-level symlink traversal.

**Prior partial fix (PR #762):** Added `voice-shared-references` to the `root_dirs` allowlist so the stale-cleanup loop preserves its symlink. This prevented the SYMLINK from being removed but did not prevent operations that traverse INTO the symlinked directory from deleting repo files.

**Fix:** Added `_resolves_inside(path, repo_root)` guard function. Applied before every `rmtree`, `unlink`, and `rmdir` in:
- `_ensure_symlink` (all destructive branches including race recovery)
- `_sync_skills_flat_symlinks` stale-cleanup loop
- `_main_inner` deferred stale-cleanup loop

When the guard detects that a path resolves inside the repo root, the operation is skipped with a loud stderr warning. Fail-safe: skip + warn beats guessing.

**Tests:** 9 new tests in `TestResolvesInside` (5) and `TestRepoSideDeletionGuard` (4). Key test: `test_ensure_symlink_blocks_rmtree_through_parent_symlink` — simulates `~/.claude/skills -> repo/skills/` and verifies `_ensure_symlink` refuses to rmtree the repo directory.

**Lesson:** Symlinks from the mirror (`~/.claude/`) into the repo create a trust-boundary violation. `Path.is_symlink()` only checks the final component; parent-level symlinks are invisible to it. Defense-in-depth: resolve the full path and check against the repo root before any destructive operation, regardless of which code path reaches the deletion.
